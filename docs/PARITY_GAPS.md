# Parity Gaps — authoritative exact-clone gap list

This is the authoritative, exact-clone gap list for the Shhhcribble Windows app, derived from a multi-agent adversarial audit of the macOS Swift source (the behavioural ground truth). Each agent re-read both sides line by line — the macOS source plus its call sites, and the Windows port plus its callers — and verified every quoted value, line number, RGB, timing, and string against the actual files. The findings below are the consolidated, deduplicated set of every place where the Windows app **differs from** or is **missing** behaviour present on macOS, plus the features deliberately **cut** for Windows. Implement the table below verbatim and the Windows app becomes a 1:1 behavioural clone of the macOS original (modulo the documented platform substitutions). Values, strings, and timings are quoted exactly so an implementer needs nothing beyond this document.

Severity legend: **HIGH** = behavioural defect (re-entry, dropped audio, stuck state, broken contract). **MEDIUM** = real UX/behaviour gap with no crash. **LOW** = cosmetic, diagnostic, or a justified/documented platform deviation.

## Implementation plan — grouped by Windows file (edit in this order so shared contracts land before their callers)

Order rationale: define new contracts on the leaf subsystems (MicRecorder level/onReady/onError, LozengeWindow new methods, MusicPauser delayed resume, SettingsStore additions, ParakeetModels strings) FIRST, then wire their callers in AppController, then do the self-contained XAML/cosmetic edits last. AppController is touched by many findings; do all its edits in one pass after the contracts exist.

### 1. `src/Shhhcribble.Core/Transcription/ParakeetModels.cs`  (no callers depend on changing first)
- Set `V3.DisplayName = "Parakeet V3 (~494 MB) – Multilingual ✦"` and `V2.DisplayName = "Parakeet V2 (~476 MB) – English-optimized"` (en-dash U+2013, ✦ on V3). Leave the archive URLs unchanged.

### 2. `src/Shhhcribble.Core/Transcription/ParakeetTranscriptionEngine.cs`
- Change the min-audio guard to `if (samples16k.Length <= WavAudio.TargetSampleRate / 2) return "";` (skip at exactly 8000 samples, matching macOS `> minSamples`).

### 3. `src/Shhhcribble.Windows/Audio/MicRecorder.cs`  (defines: level callback, onReady, onError, currentSamples — all consumed by AppController)
- Add `public event Action<float>? LevelChanged;` (or `Start(Action<float>)`). In `OnData`, after building mono frames, compute `rms = sqrt(sumOfSquares / max(frameCount,1))`, `level = Math.Min(rms * 20.0f, 1.0f)`, raise it (gain 20.0f, clamp 1.0f exact). Marshal to UI thread at the AppController layer.
- Add `Action? onReady` to `Start()` + a `bool _didFireReady` latch (reset in Start/Stop). Poll capture-device readiness on a ~25 ms timer up to ~1200 ms (MMDevice channel count or first non-silent buffer); on first ready lock + `_mono.Clear()` (discard pre-roll) then invoke `onReady` once on the UI thread; if the 1200 ms budget elapses invoke anyway.
- Add `Action<string>? onError`. Validate `_format.SampleRate > 0 && _format.Channels > 0` after reading WaveFormat → else `onError("Microphone not ready — try again")` and abort. Wrap WasapiCapture construction + `StartRecording()` in try/catch → on failure dispose + `onError("Couldn't start microphone")`.
- Add a thread-safe `CurrentSamples` snapshot (resampled to 16 kHz under `_lock`) — only if live preview is brought into scope.
- Implement `IMMNotificationClient` via `MMDeviceEnumerator.RegisterEndpointNotificationCallback`; on `OnDefaultDeviceChanged` (Capture) while recording: snapshot `_mono` under lock, dispose old capture, create new on the new default device, restore samples, re-wire `OnData`/`RecordingStopped`, `StartRecording()`. No debounce.
